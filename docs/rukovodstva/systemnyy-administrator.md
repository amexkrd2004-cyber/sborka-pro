# Руководство: системный администратор

Этот документ позволяет **с нуля развернуть и настроить** экосистему **СборкаПро**: сервер, база данных, переменные окружения, webhook **МойСклад**, сборка мобильного приложения и публикация веб-панели.  
Предполагается знакомство с командной строкой, SSH (если VPS), DNS и HTTPS на базовом уровне.

Интеграция **amoCRM ↔ МойСклад (amgroup)** к моменту внедрения **уже работает** — здесь не описывается.

Плейсхолдеры `<...>` заменяйте на ваши значения.

---

## 1. Архитектура (кратко)

```
МойСклад ──webhook──► Сервер Node.js (Express)
                           │
                           ├── БД (SQLite / PostgreSQL)
                           ├── REST API для приложения и панели
                           └── Expo Push → Android-клиенты

Кладовщик: Android APK ◄──HTTPS──► Сервер

Администратор: браузер ◄──HTTPS──► Веб-панель (React SPA)
amoCRM ◄── amgroup ◄── МойСклад  (уже настроено у вас)
```

---

## 2. Требования

| Компонент | Минимум |
|-----------|---------|
| Хостинг сервера | Railway, Render, свой VPS (Linux), и т.п. с **публичным HTTPS** |
| Node.js | Версия по `package.json` проекта (уточнить при появлении репозитория) |
| БД | SQLite для старта; PostgreSQL для продакшена с бэкапами |
| Домен | Желательно собственный + TLS; допустим выданный хостингом URL |
| Учёт МойСклад | Права API: заказы, товары, склады, смена статусов |
| Сборка APK | Учётная запись **Expo (EAS)**, Android SDK при локальной сборке (опционально) |

## 4. Переменные окружения (сервер)

Файл **`.env`** в каталоге сервера (**не коммитить** в Git). Имена из ТЗ:

| Переменная | Назначение |
|------------|------------|
| `MOYSKLAD_TOKEN` | API-токен МойСклад |
| `MOYSKLAD_ACCOUNT_ID` | ID аккаунта МойСклад (если требуется API) |
| `JWT_SECRET` | Секрет подписи JWT, **≥ 32 символов**, случайная строка |
| `PORT` | Порт процесса (например `3000`; на PaaS часто задаётся платформой) |
| `DATABASE_URL` | PostgreSQL: на Railway в сервисе приложения — публичная строка (**`DATABASE_PUBLIC_URL`** из Postgres) или Variable Reference; в локальном `.env` для **`npm run db:seed`** — тоже публичный URL (не `*.railway.internal`). SQLite — путь/имя файла, если выберете SQLite. |
| `WEBHOOK_SECRET` | Секрет проверки подлинности webhook (если реализован на сервере) |

Документация API МойСклад: [https://dev.moysklad.ru/doc/api/remap/1.2/](https://dev.moysklad.ru/doc/api/remap/1.2/)

### 4.1. Генерация `JWT_SECRET` (пример)

В Linux/macOS:

```bash
openssl rand -base64 32
```

В PowerShell можно использовать:

```powershell
[Convert]::ToBase64String((1..32 | ForEach-Object { Get-Random -Maximum 256 }))
```

## 5. База данных

1. Создать пустую БД (PostgreSQL) **или** убедиться, что путь к файлу SQLite доступен для записи процессу.  
2. Схема таблиц применяется при старте сервера (`initDb`) и при **`npm run db:seed`**.  
3. Первого **администратора**: **`npm run db:seed`** из каталога `server/` (см. **`docs/railway-sborka-pro.md`** — шаг 7). На Railway для сида с домашнего ПК в `.env` нужен **`DATABASE_PUBLIC_URL`**, не внутренний хост.

**Резервное копирование:** для PostgreSQL — регулярный `pg_dump` или автобэкап хостинга; для SQLite — копирование файла на отдельный носитель по расписанию.

## 6. Запуск сервера

### Локально (разработка)

Из корня проекта:

```bash
cd server
npm install
copy .env.example .env
npm run dev
```

(В PowerShell копирование: `Copy-Item .env.example .env`.) Сервер слушает порт **3000** или значение `PORT` из `.env`. Подробности: **`server/README.md`**.

- Проверка: `GET http://127.0.0.1:3000/health` — ответ `{"ok":true,"service":"sborka-pro-server",...}`.

На **Railway / Render**: подключить репозиторий, корень сервиса — каталог `server/`, команда старта `npm start`, переменные окружения задать в панели (пока достаточно `PORT`, если платформа задаёт его сама).

**Railway — пошагово (новый проект, GitHub, поле Root Directory = `server`, домен, вебхук):** отдельный файл **[`docs/railway-sborka-pro.md`](../railway-sborka-pro.md)**.

### Продакшен

- `GET https://<ваш-хост>/health`  
- Логи без утечки `MOYSKLAD_TOKEN` и паролей.

## 7. Webhook МойСклад

В интерфейсе МойСклад в блоке **Обмен данными** обычно есть только **Токены** — отдельного меню «Вебхуки» для JSON API может не быть. **Вебхуки регистрируются через JSON API:** `POST https://api.moysklad.ru/api/remap/1.2/entity/webhook` с заголовком `Authorization: Bearer <токен>` и телом, например:

```json
{
  "url": "https://<ваш-хост>/webhook/moysklad",
  "action": "UPDATE",
  "entityType": "customerorder"
}
```

Полная схема (два вебхука, формат `events`, переменные Railway, догрузка заказа): **[`docs/moysklad-integration.md`](../moysklad-integration.md)**. Пошаговое создание вебхуков: **[`docs/railway-sborka-pro.md`](../railway-sborka-pro.md)** — **«Шаг 6»** (там же: **`npm run moysklad:register-webhooks`** в `server/` и **`curl.exe`**, если из PowerShell приходит **415** на `Invoke-RestMethod`).

Если позже настроите проверку подписи — сохраните секрет в `WEBHOOK_SECRET` на сервере.

На сервере (после доработки логики) фильтрация: реагировать на переход в статус **«Сборка»** — имя статуса должно совпасть с МойСклад.

**Проверка:** изменить тестовый заказ покупателя → в логах Railway строка `[webhook/moysklad]`.

> **Скриншот:** экран создания вебхука в МойСклад (без полного URL с секретами в публичных инструкциях по желанию замазать).

## 8. Настройки в МойСклад (согласование с ТЗ)

Системный администратор **контролирует**, что сделано по инструкции из ТЗ (детали — у функционального заказчика):

- **Дополнительные поля заказа покупателя:** тип доставки, ТК, трек, примечание для сборщика и т.д.  
- **Статусы заказа:** Новый, Подтверждён, **Сборка**, **Собран**, **Отгружен**, отказ/отмена — как в регламенте компании.

Имена статусов в API должны **совпадать** с тем, что ожидает код сервера (зафиксировать в `sborka.md` после первой интеграции).

## 9. Мобильное приложение (Expo / EAS)

### 9.1. Переменные для клиента

В `app.config.js` / `eas.json` / `.env` Expo (публичные URL можно префиксом `EXPO_PUBLIC_`):

| Переменная | Пример |
|------------|--------|
| `EXPO_PUBLIC_API_URL` | `https://your-app.railway.app` |

### 9.2. Сборка APK

Нативные модули (сканер, принтер) требуют **development build**, не обычный Expo Go.

Типовой путь:

```bash
cd mobile
npm ci
npx eas login
npx eas project:init
npx eas build --platform android --profile development
```

В проекте есть базовый `mobile/eas.json` с профилями `development`, `preview`, `production`.

> Важно: push-уведомления через `expo-notifications` на Android в **Expo Go** не работают.  
> Проверять push только в development build / APK.

### 9.3. Распространение APK

- Загрузка файла на корпоративный портал / облако.  
- Или внутренний каталог устройств (MDM).  
- Публикация в Google Play — отдельная процедура (если потребуется).

> **Скриншот:** страница успешной сборки в Expo Dashboard.  
> ![EAS Build](screenshots/sysadmin-02-eas-build.png)

Документация Expo: [https://docs.expo.dev/build/introduction/](https://docs.expo.dev/build/introduction/)

## 10. Веб-панель администратора

1. Собрать статику: `npm run build` в каталоге `web/`.  
2. Раздавать через тот же Express (`express.static`) **или** отдельный nginx/ CDN — как решено в проекте.  
3. Убедиться, что маршруты SPA перенаправляются на `index.html` (history API).

Базовый URL сообщите администраторам (раздел 2 руководства администратора).

---

## 11. TLS и домен

- Использовать **HTTPS** везде (МойСклад webhook не должен упираться в самоподписанный сертификат без исключений).  
- На VPS: Let's Encrypt (certbot), обновление сертификатов по cron.

---

## 12. Мониторинг и логи

- Включить ротацию логов на сервере.  
- Настроить алерт при падении процесса (systemd, Docker healthcheck, встроенные средства PaaS).  
- Не логировать тела запросов с токенами и паролями.

---

## 13. Обновление системы (релиз)

1. Остановить сервер (или rolling deploy на PaaS).  
2. `git pull`, `npm ci`, миграции БД, `npm run build`.  
3. Запуск, smoke-тест: логин, список заказов, тестовый webhook.  
4. При изменении API — выпустить новую **сборку APK** и уведомить склад.

Записать версию и дату в `sborka.md` (журнал изменений).

## 14. Чеклист «с нуля до первого кладовщика»

- [ ] Сервер доступен по HTTPS.  
- [ ] БД создана, миграции применены.  
- [ ] `.env` заполнен, сервис стартует без ошибок.  
- [ ] Webhook МойСклад создан, тестовый заказ даёт запись в логах.  
- [ ] Создан кладовщик в БД, назначены склады.  
- [ ] Собран APK, `EXPO_PUBLIC_API_URL` указывает на прод.  
- [ ] Кладовщик установил APK, вошёл, видит тестовый заказ.  
- [ ] Администратор открыл панель и видит дашборд.

---

## 15. Восстановление после сбоя

1. Поднять новый инстанс сервера с тем же **Node** и переменными `.env`.  
2. Восстановить БД из последнего **бэкапа**.  
3. Проверить webhook URL в МойСклад (если сменился домен).  
4. Пересобрать APK **только если** изменился публичный URL API.
