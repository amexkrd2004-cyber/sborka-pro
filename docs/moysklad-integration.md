# Интеграция СборкаПро ↔ МойСклад (рабочая схема)

Документ фиксирует **конкретную** реализацию по [JSON API 1.2](https://dev.moysklad.ru/doc/api/remap/1.2/) и [сущности «Вебхуки»](https://github.com/moysklad/api-remap-1.2-doc/blob/master/md/dictionaries/_webhook.md). Платный тариф и роль администратора — условия МойСклад.

---

## 1. Две роли токена

| Что | Зачем |
|-----|--------|
| **Токен JSON API** (`MOYSKLAD_TOKEN`) | Хранится **только на сервере** (Railway Variables). Нужен для **создания вебхуков** через API и для **запросов к заказу** после уведомления. |
| **Входящий вебхук** | МойСклад сам шлёт **POST** на ваш URL. В теле **нет** вашего токена — только описание события. Сервер по `href` из уведомления **сам** запрашивает заказ с заголовком `Authorization: Bearer <токен>`. |

Итого: токен не «приезжает во вебхуке»; он используется **сервером при обращении к API** после приёма уведомления.

---

## 2. Какие вебхуки регистрируем (оптимально — два)

Пара **`entityType` + `action`** уникальна для каждого вебхука. Для заказа покупателя тип сущности в API: **`customerorder`**.

| № | `entityType` | `action` | Зачем |
|---|----------------|----------|--------|
| 1 | `customerorder` | **UPDATE** | Смена статуса на «Сборка», правки состава, полей — основной сценарий для кладовщика. |
| 2 | `customerorder` | **CREATE** | Новый заказ, если он сразу создаётся в «Сборке» (или вы хотите ловить появление заказа в любом статусе и фильтровать по API). |

**Минимум для ТЗ** (только переход в сборку из другого статуса): теоретически хватило бы **UPDATE**, но на практике заказ может **создаться** уже в «Сборке» — тогда без **CREATE** уведомление не придёт. Поэтому в продакшене держим **оба** вебхука на один и тот же URL.

**Один URL для обоих:**  
`https://<ваш-хост>/webhook/moysklad`

Регистрация (пример тела):

```json
{
  "url": "https://<ваш-хост>/webhook/moysklad",
  "action": "UPDATE",
  "entityType": "customerorder"
}
```

```json
{
  "url": "https://<ваш-хост>/webhook/moysklad",
  "action": "CREATE",
  "entityType": "customerorder"
}
```

Дополнительно (не обязательно на старте): для `UPDATE` в API можно указать **`diffType": "FIELDS"`** — тогда в событии может быть массив **`updatedFields`** (какие поля изменились). Для первой версии достаточно **всегда дёрнуть GET заказа по `href`** и смотреть текущий статус — проще и устойчивее.

## 3. Что именно приходит во входящем POST (вам разбирать не нужно — делает сервер)

Формат из официальной документации:

- Запрос: **POST** `https://ваш-сервер/webhook/moysklad?requestId=<uuid>`
- Заголовок **User-Agent** содержит `MoySklad webhook touch agent` (можно использовать для грубой фильтрации спама).
- Тело JSON:

```json
{
  "auditContext": {
    "meta": { "type": "audit", "href": "https://api.moysklad.ru/api/remap/1.2/audit/..." },
    "moment": "2021-07-21 15:51:16",
    "uid": "login@..."
  },
  "events": [
    {
      "meta": {
        "type": "customerorder",
        "href": "https://api.moysklad.ru/api/remap/1.2/entity/customerorder/<id>"
      },
      "action": "UPDATE",
      "accountId": "...",
      "updatedFields": ["state"]
    }
  ]
}
```

- В **`events`** может быть **несколько** элементов — обрабатываем все, где `meta.type === "customerorder"`.
- **`requestId`** повторяется при ретраях МойСклад — на фазе B можно кешировать для идемпотентности.

## 4. Ограничение 1,5 секунды и оптимальный пайплайн

МойСклад ждёт ответ **200 или 204** примерно **1500 ms**. Иначе — повторы (до 3).

**Правильная схема:**

1. Принять JSON, быстро провалидировать форму.
2. Сразу ответить **`200`** с телом `{ "ok": true }` (как сейчас).
3. **После** отправки ответа (асинхронно): для каждого события `customerorder` выполнить **GET** по `meta.href` с `Bearer`-токеном, прочитать **`state.name`** (или иной признак статуса в вашем аккаунте).
4. Если статус совпадает с целевым (**по умолчанию имя статуса «Сборка»**, задаётся переменной `MOYSKLAD_ASSEMBLY_STATE_NAME`) — дальше логика ТЗ: очередь, push, БД (фаза B).

Тяжёлую работу никогда не держим **до** `res.send`.

---

## 5. Распознавание статуса «Сборка»

В API у заказа покупателя есть объект **`state`** (текущий статус документа). Имя статуса в интерфейсе МойСклад должно **в точности** совпадать с тем, что приходит в **`state.name`** (регистр и пробелы — как в вашем справочнике статусов).

Переменная окружения:

- **`MOYSKLAD_ASSEMBLY_STATE_NAME`** — например `Сборка` (значение по умолчанию в коде можно задать так же).

Если у вас несколько статусов должны открывать сборку — на фазе B вводим список через запятую или внешнюю настройку.

---

## 6. Поток данных (сводка)

```text
МойСклад (изменение заказа)
    → POST /webhook/moysklad?requestId=…  (тело: events + auditContext)
Сервер СборкаПро
    → 200 OK (сразу)
    → GET meta.href  (Authorization: Bearer MOYSKLAD_TOKEN)
    → если state.name == MOYSKLAD_ASSEMBLY_STATE_NAME → дальше бизнес-логика (push, учёт)
```

## 7. Где что лежит в репозитории

| Файл | Назначение |
|------|------------|
| `server/src/routes/webhook.js` | Приём POST, быстрый ответ, запуск асинхронной обработки. |
| `server/src/services/moyskladWebhookWorker.js` | Разбор `events`, запрос заказа, логика «попал ли в Сборку». |
| `server/src/services/moyskladApi.js` | Минимальный клиент GET по `href`. |
| `docs/railway-sborka-pro.md` | Шаг 6: как создать вебхуки через API. |
| `sborka.md` | План фаз; при смене статусов/полей — журнал. |
