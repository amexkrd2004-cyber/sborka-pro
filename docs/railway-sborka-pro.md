# Пошаговое развёртывание СборкаПро (backend) на Railway

Инструкция для **монорепозитория**: в корне лежат `sborka.md`, `docs/`, папка **`server/`** с Node.js-приложением. Railway должен собирать и запускать **только `server/`**, иначе сборка упадёт (в корне нет `package.json`).

У вас уже есть аккаунт Railway — войдите на [https://railway.app](https://railway.app).

---

## Что получится в конце

- Публичный URL вида `https://sborka-pro-production-xxxx.up.railway.app` (имя будет своё).
- Рабочие адреса:
  - `GET …/health` — проверка, как у вас локально (`{"ok":true,"service":"sborka-pro-server","phase":"A"}`).
  - `POST …/webhook/moysklad` — endpoint для МойСклад.

**Файлы никуда «вручную на сервер копировать» не нужно** — Railway забирает код из Git (GitHub/GitLab/Bitbucket) и сам ставит зависимости. Локально вы только пушите коммиты в репозиторий.

### Windows: частые ошибки в терминале

1. **Скопировали строки `` ``` `` или `` ```powershell ``** — это разметка документа, а не команда. PowerShell выдаст ошибку вроде «имя \`\`\`powershell не распознано». **Вставляйте только текст команд** из блока (например одну строку `git status`), без обрамления кавычками.
2. **«Имя git не распознано»** — Git не установлен или не в PATH. Установите [Git for Windows](https://git-scm.com/download/win), в мастере выберите вариант, чтобы Git был доступен из командной строки, **закройте и снова откройте** терминал, проверьте: `git --version`.
3. **`Author identity unknown` / `Please tell me who you are`** — Git ещё не настроен. Один раз выполните (подставьте **свои** имя и почту; email лучше как в аккаунте GitHub или [приватный noreply](https://docs.github.com/en/account-and-profile/setting-up-and-managing-your-personal-account-on-github/managing-email-preferences/setting-your-commit-email-address)):

   ```powershell
   git config --global user.name "Ваше Имя"
   git config --global user.email "your.email@example.com"
   ```

   Затем снова: `git commit -m "..."`.

## Шаг 0. Подготовить Git (если ещё не сделано)

Выполняется **один раз** на вашем ПК в папке проекта (например `c:\Sborka_Project`).

1. Откройте терминал в корне проекта.
2. Проверьте, есть ли уже Git:

   ```powershell
   git status
   ```

   Если пишет «not a git repository» — инициализируйте:

   ```powershell
   git init
   ```

3. Убедитесь, что **не попадут** в репозиторий `node_modules` и секреты. В корне должен быть файл `.gitignore` (у вас уже есть записи для `server/node_modules/` и `server/.env`).

4. Добавьте файлы и сделайте первый коммит:

   ```powershell
   git add .
   git commit -m "Sborka Pro: server phase A, docs"
   ```

5. **Создайте репозиторий на GitHub** — см. блок ниже («зачем», «почему пустой», «как в интерфейсе»).

6. Привяжите удалённый репозиторий и отправьте код (подставьте свой URL):

   ```powershell
   git remote add origin https://github.com/<ваш-логин>/sborka-pro.git
   git branch -M main
   git push -u origin main
   ```

**Проверка, что привязка и push прошли успешно**

1. **Список удалённых репозиториев и URL** (должен быть `origin` и ваш адрес на GitHub):

   ```powershell
   git remote -v
   ```

   Ожидается две строки: `origin  https://github.com/...` для `fetch` и для `push`.

2. **Связь ветки с GitHub** (после `git push -u origin main`):

   ```powershell
   git branch -vv
   ```

   У ветки `main` в скобках должно быть что-то вроде `[origin/main]` — значит «вышка» отслеживается.

3. **Состояние относительно сервера:**

   ```powershell
   git status
   ```

   Если всё отправлено и локально нет незакоммиченных правок: *«Your branch is up to date with 'origin/main'»* (или русская формулировка той же мысли).

4. **Доступ к GitHub по сети** (опционально; запросит логин/токен, если нужна авторизация):

   ```powershell
   git ls-remote origin
   ```

   Если команда выводит хеши веток — соединение с `origin` работает.

5. **В браузере:** откройте страницу репозитория на GitHub — должны быть видны папки `server/`, `docs/`, файлы `README.md`, `sborka.md` и т.д. (как у вас в проекте после коммита).

Если `git remote add` выдал ошибку *«remote origin already exists»*, сначала посмотрите `git remote -v`. Чтобы **сменить** URL: `git remote set-url origin https://github.com/<логин>/<репо>.git`.

Дальше любые изменения: `git add` → `git commit` → `git push` — Railway подхватит новый деплой сам (если включён auto-deploy от ветки `main`).

### К шагу 5: зачем GitHub и что значит «пустой репозиторий»

**Для чего это Railway и вам**

- **Railway не забирает папку с вашего диска напрямую** (в типовом сценарии «Deploy from GitHub»). Он подключается к **удалённому** хранилищу кода — чаще всего **GitHub** — и при каждом `git push` заново собирает и деплоит проект.
- Репозиторий на GitHub — это **центральная копия** проекта в интернете: с вашего ПК вы туда **отправляете** изменения (`git push`), Railway **читает** оттуда же. Так код оказывается на сервере без ручной «заливки файлов» через сайт.

**Почему советуют создать репозиторий «пустым» (без README и без лишних галочек)**

- На GitHub при создании репозитория можно сразу добавить **README** или **.gitignore** — тогда GitHub сам сделает **первый коммит** на своей стороне.
- У вас на компьютере уже есть **свой** первый коммит (шаг 4). Получаются **две разные «стартовые» истории**, и при первом `git push` Git может отказаться или попросить слить истории — для новичка это лишняя путаница.
- **Пустой** репозиторий = на GitHub **ещё нет ни одного коммита**. Тогда ваш первый `git push` просто **загружает** всё, что вы уже закоммитили локально, без конфликтов.

У вас в проекте уже есть свой `README.md` и `.gitignore` в папке — **их не нужно дублировать** галочками на GitHub при создании репозитория; они попадут на GitHub вместе с `git push`.

**Пошагово в браузере (github.com)**

1. Войдите в аккаунт GitHub.
2. Нажмите **«+»** (правый верхний угол) → **New repository** (новый репозиторий).
3. **Repository name:** например `sborka-pro` (латиница, без пробелов; имя может быть любым, главное — запомнить его для Railway).
4. Выберите **Public** или **Private** (для Railway подходит и тот и другой, если Railway имеет доступ к репозиторию).
5. **Не отмечайте:**
   - **Add a README file**
   - **Add .gitignore**
   - **Choose a license**  
   (все три должны быть **выключены**, чтобы репозиторий остался без автоматического первого коммита.)
6. Нажмите **Create repository**.

Откроется страница с подсказками Git — для вашей ситуации используйте **шаг 6** выше (`git remote add` … `git push`), а не команды «initialize with README» с этой страницы.

**Если репозиторий уже создан с README на GitHub**

Тогда истории разошлись; проще всего спросить в чате или выполнить один раз слияние (например `git pull origin main --allow-unrelated-histories` с последующим коммитом и снова `git push`). Для первого раза проще **удалить** такой репозиторий на GitHub и создать новый **пустой**, как в пунктах 1–6.

## Шаг 1. Новый проект на Railway

1. Зайдите на [railway.app](https://railway.app) → **Dashboard**.
2. Нажмите **New Project** (или **+ New Project**).
3. Выберите **Deploy from GitHub repo** (или GitLab — как подключён аккаунт).
4. Если попросит — **разрешите Railway доступ** к репозиториям и выберите репозиторий **`sborka-pro`** (или как вы его назвали).
5. Railway создаст **проект** и один **деплой** (в новом UI отдельного слова *Service* может не быть). Узел приложения — это **карточка слева** с именем (например `sborka-pro`) и статусом Online, либо плитка на схеме проекта — по ней открываются вкладки **Deployments**, **Settings**, **Variables**.

## Шаг 2. Указать корень сборки — папка `server` (обязательно)

Сейчас код лежит не в корне репозитория, а в **`server/`**. Без этого шага сборка часто завершается ошибкой.

**Если Railway сам показал диагноз** вроде *«Set the root directory to 'server'»* и кнопку **Set root directory** — нажмите её: это как раз нужная настройка, искать пункт *Service* вручную не обязательно.

Вручную:

1. Откройте ваш **проект** на Railway.
2. Откройте **сервис** — карточку/узел с именем репозитория или сервиса (см. п.5 шага 1), не только заголовок проекта.
3. Перейдите в **Settings**.
4. Найдите блок **Root Directory** (или **Source → Root Directory**).
5. Впишите: **`server`** (без слэша в начале, без `package.json`).
6. Сохраните (**Save**). Railway обычно **перезапускает деплой** автоматически.

Проверка в логах деплоя (**Deployments → View logs**):

- должны быть строки вроде `npm install` / `npm ci`;
- старт: `npm start` или `node src/index.js`;
- в логах приложения: `Sborka Pro server listening on...`

## Шаг 6. Подключить МойСклад (вебхук JSON API)

### Почему в интерфейсе нет пункта «Вебхуки»

В разделе **Настройки → Обмен данными → Токены** создаётся только **токен доступа к JSON API**. Это правильное место для шага «получить токен», но **список и создание вебхуков там не живут**.

По [документации МойСклад](https://dev.moysklad.ru/doc/api/remap/1.2/workbook/#/workbook/vebhuki): **работа с вебхуками возможна только через JSON API** (создание — запрос `POST …/entity/webhook`). Отдельной страницы «Настройки → Вебхуки» в духе старых инструкций в текущем UI может не быть.

**Ограничения (из той же документации):**

- вебхуки доступны **на платном тарифе** (на бесплатном — уточните в поддержке МойСклад);
- создавать/смотреть вебхуки может **администратор аккаунта**;
- МойСклад ждёт ответ **200 или 204 в течение ~1,5 с**; при «холодном» старте Railway первый вызов иногда обрывается — при пустых логах повторите изменение заказа.

**Иное:** в интерфейсе есть **Сценарии** с действием «отправить вебхук» ([справка](https://support.moysklad.ru/hc/ru/articles/360002296078)) — это **другой** механизм (часто GET с подстановками `{id}` и `{type}`). Наш сервер принимает **POST** в формате уведомлений JSON API, поэтому для СборкаПро нужен именно **вебхук через API** ниже.

### Порядок действий (упрощённый чеклист)

Подробности «как устроено внутри»: **[`docs/moysklad-integration.md`](moysklad-integration.md)**.

**Зачем вообще «адрес в интернете» (тот самый https://…)**  
МойСклад работает **в облаке**. Когда меняется заказ, он должен **вызвать ваш сервер по ссылке**, как вы набираете сайт в браузере. Адрес **`http://127.0.0.1`** или «мой компьютер» из интернета **недоступен** — до него МойСклад не дозвонится. Поэтому сервер сначала выкладывают на **Railway** и получают **одну постоянную ссылку** вида `https://что-то.up.railway.app`. К этой ссылке мы просто **дописываем хвост** `/webhook/moysklad` — это «номер квартиры» на том же доме, куда слать уведомления. **Отдельный третий адрес не придумывается** — это тот же Railway-URL, что вы уже открывали для проверки `/health`, только с другим окончанием.

---

Сделайте по шагам **по порядку**:

**А.** В МойСклад: **Настройки → Обмен данными → Токены** — создайте токен, **сразу скопируйте** (после закрытия окна его не покажут).

**Б.** В Railway: **Variables** → добавьте **`MOYSKLAD_TOKEN`** = вставьте скопированный токен.  
*(Без этого сервер увидит «стук» от МойСклад, но не сможет прочитать заказ.)*

**В.** Возьмите **ровно тот адрес**, который выдаёт Railway (тот же, где проверяли `/health`).  
Пример: если в браузере открывается  
`https://sborka-pro-production-xxxx.up.railway.app/health`,  
то для МойСклад нужна строка **на том же домене**:  
`https://sborka-pro-production-xxxx.up.railway.app/webhook/moysklad`  
Скопируйте её в блокнот — это значение для переменной `$myServer` ниже.

**Г.** Зарегистрировать **два** вебхука в МойСклад (CREATE и UPDATE для `customerorder`). Нужны те же **токен** и **полный URL** `https://…/webhook/moysklad`, что в пунктах **А** и **В**.

Из-за особенностей **Windows PowerShell 5.x** запрос `Invoke-RestMethod` к API МойСклад иногда всё равно даёт **415 Unsupported Media Type** (неверно уходит `Content-Type` или кодировка тела). Надёжнее один из вариантов ниже.

---

**Вариант Г1 (рекомендуем): Node.js из папки `server`** — тот же стек, что у проекта:

```powershell
cd c:\Sborka_Project\server
npm install
```

В `.env` (или только в командной строке на один раз) задайте **`MOYSKLAD_TOKEN`** и **`MOYSKLAD_WEBHOOK_URL`** = полная ссылка на вебхук, например `https://sborka-pro-production.up.railway.app/webhook/moysklad`.

```powershell
npm run moysklad:register-webhooks
```

Либо без `.env`:  
`node scripts/register-moysklad-webhooks.js --token "ВАШ_ТОКЕН" --url "https://….up.railway.app/webhook/moysklad"`

При успехе в консоли будут строки `UPDATE: OK` и `CREATE: OK`. Ошибка HTTP (например, дубликат вебхука) скрипт выведет целиком и завершится с ненулевым кодом.

---

**Вариант Г2: `curl.exe` из PowerShell** (настоящий curl из Windows; команда **`curl` без `.exe`** в PowerShell — это **другой** cmdlet, не подходит):

```powershell
$token = "ВАШ_ТОКЕН"
$myServer = "https://ваш-поддомен.up.railway.app/webhook/moysklad"

foreach ($action in @("UPDATE", "CREATE")) {
  $json = @{ url = $myServer; action = $action; entityType = "customerorder" } | ConvertTo-Json -Compress
  $tmp = Join-Path $env:TEMP ("moysklad-webhook-" + $action + ".json")
  [IO.File]::WriteAllText($tmp, $json, [Text.UTF8Encoding]::new($false))
  curl.exe -fS --compressed -X POST "https://api.moysklad.ru/api/remap/1.2/entity/webhook" `
    -H "Authorization: Bearer $token" `
    -H "Content-Type: application/json" `
    -d "@$tmp"
  Remove-Item $tmp -ErrorAction SilentlyContinue
}
Write-Host "Готово (если curl не ругался — оба вебхука созданы)."
```

Флаг **`-f`** у `curl.exe` завершает команду с ошибкой, если HTTP не 2xx — сообщение «Готово» тогда не будет обманчивым.

---

**Вариант Г3: только PowerShell 5.x** — если Г1/Г2 неудобны, можно снова пробовать `Invoke-RestMethod` с UTF-8-байтами (как в версии 1.3 документа); на части систем это по-прежнему даёт **415** — тогда обязательно **Г1** или **Г2**.

Что делают команды: говорят МойСклад «при **новых** заказах и при **изменениях** заказа шли уведомления на ваш **https://…/webhook/moysklad**». По правилам API это **две** отдельные регистрации (CREATE и UPDATE).

**Д.** (Необязательно.) Проверить список: в браузере или через PowerShell — `GET https://api.moysklad.ru/api/remap/1.2/entity/webhook` с заголовком `Authorization: Bearer <токен>`.

**Е.** В МойСклад измените тестовый заказ. В логах Railway должны появиться `[webhook/moysklad]` и `[moysklad-worker]`.

Официальное описание: [Workbook → Вебхуки](https://dev.moysklad.ru/doc/api/remap/1.2/workbook/#/workbook/vebhuki), тело входящего уведомления — [сущность Webhook в справочнике API](https://github.com/moysklad/api-remap-1.2-doc/blob/master/md/dictionaries/_webhook.md).

## Шаг 7. PostgreSQL, JWT и первый пользователь (фаза B) — для новичков

Здесь не «программирование», а **настройка Railway и переменных**, чтобы сервер мог хранить пользователей и выдавать токены. Код уже в репозитории; вам нужно **создать базу**, **прописать секреты** и **один раз создать администратора** (сид).

**Зачем это:** без `DATABASE_URL` сервер работает только как **фаза A** (вебхук МойСклад). С базой и JWT появляются вход **`POST /auth/login`**, список заказов **`GET /orders`**, захват заказа и т.д. Подробнее об эндпоинтах: **`server/README.md`**.

### 7.1. Что такое `DATABASE_URL` простыми словами

Это **одна длинная строка-пароль** вида `postgresql://логин:пароль@хост:порт/имя_базы` — по ней программа подключается к PostgreSQL так же, как вы заходите на сайт по адресу. Её выдаёт Railway для созданной базы; вы **копируете** её в переменные **сервиса с приложением** (не публикуйте в чат и не коммитьте в Git).

### 7.2. Создать PostgreSQL в том же проекте Railway

1. Зайдите на [railway.app](https://railway.app), откройте **тот же проект**, где уже крутится **sborka-pro** (или как у вас называется сервис с Node).
2. Нажмите **«New»** (или **«+ Create»**) → выберите **«Database»** → **«Add PostgreSQL»** (или аналог «PostgreSQL»).
3. Подождите, пока база **развернётся** (статус станет зелёным / Active).
4. Откройте **карточку именно сервиса PostgreSQL** (слева в списке сервисов появится отдельная плитка «Postgres», не путайте с приложением).

### 7.3. Передать `DATABASE_URL` в сервис приложения (Node)

У приложения и у базы — **разные** сервисы. Переменная с строкой подключения должна оказаться у **приложения**, иначе Node её не увидит.

**Способ A (рекомендуем — публичный URL, работает и в облаке, и для сида с ПК):**

1. Откройте сервис **PostgreSQL** → вкладка **Variables**.
2. Найдите **`DATABASE_PUBLIC_URL`** — это готовая строка `postgresql://…` на **внешний** хост Railway. Покажите значение и **скопируйте** целиком.  
   *(Если `DATABASE_PUBLIC_URL` нет — включите у Postgres **TCP Proxy** и соберите URL вручную, см. шаг 7.6.)*
3. Откройте сервис **приложения СборкаПро** (Node) → **Variables** → **New Variable**.
4. Имя: **`DATABASE_URL`**. Значение: **вставьте скопированный `DATABASE_PUBLIC_URL`** **или** при добавлении переменной выберите **Variable Reference** / ссылку на **`DATABASE_PUBLIC_URL`** сервиса Postgres (в интерфейсе Railway это часто удобнее: при смене пароля базы ссылка обновится сама).
5. Сохраните.

**Способ B (только внутри Railway):** переменная **`DATABASE_URL`** у Postgres с хостом **`*.railway.internal`** подходит для связи **сервис ↔ сервис** в одном проекте, но **не** для запуска **`npm run db:seed` с домашнего компьютера**. Для сида с ПК в локальный **`server/.env`** всё равно положите **`DATABASE_PUBLIC_URL`** (или строку с TCP Proxy).

**Способ C (Connect):** если Railway при **Connect** к базе сам добавил **`DATABASE_URL`** в приложение — откройте его значение: если там `railway.internal`, для сида с ПК используйте отдельно **`DATABASE_PUBLIC_URL`** в `.env`, как в способе B.

После сохранения Railway обычно **сам перезапускает** деплой приложения — это нормально.

### 7.4. Задать `JWT_SECRET`

1. В сервисе **приложения** → **Variables** → новая переменная **`JWT_SECRET`**.
2. Значение — **любая случайная строка не короче 32 символов** (буквы и цифры). Это «мастер-ключ» подписи токенов: если его сменить, старые токены перестанут работать.

**Как сгенерировать на Windows (PowerShell), не придумывая вручную:**

```powershell
-join ((48..57) + (65..90) + (97..122) | Get-Random -Count 40 | ForEach-Object { [char]$_ })
```

Скопируйте вывод в значение **`JWT_SECRET`** в Railway.

### 7.5. Переменные для сида (один раз создать админа)

Сид — это скрипт **`npm run db:seed`**, который создаёт в базе первого пользователя с ролью **admin** (логин/пароль для входа в API).

В **Variables** сервиса приложения добавьте:

| Имя | Обязательно? | Что указать |
|-----|----------------|---------------|
| **`SEED_ADMIN_PASSWORD`** | Да (для сида) | Пароль **не короче 8 символов** — тот, которым потом войдёте в `POST /auth/login`. |
| **`SEED_ADMIN_LOGIN`** | Нет | Логин; если не задать — будет **`admin`**. |
| **`SEED_ADMIN_DISPLAY_NAME`** | Нет | Отображаемое имя в БД. |

**Важно:** после успешного сида **`SEED_ADMIN_PASSWORD`** можно **удалить** из Variables (чтобы пароль не хранился лишний раз) — пользователь уже в таблице `users`. Логин и пароль для входа вы уже задали при сиде.

### 7.6. Запустить сид `npm run db:seed` (один раз)

Сид **подключается к той же базе**, что указана в `DATABASE_URL`, и создаёт таблицы (если их ещё нет) + пользователя.

**Вариант 1 — с вашего ПК (часто проще всего):**

1. На компьютере откройте папку проекта `c:\Sborka_Project\server`.
2. В файле **`server/.env`** вставьте строку **`DATABASE_URL=…`** **без `#` в начале**. Лучше всего скопировать значение **`DATABASE_PUBLIC_URL`** из **Variables сервиса Postgres** (тот же текст, что вы уже использовали для успешного сида). Так сид с ПК подключается к базе без ручной сборки URL.  
   Если **`DATABASE_PUBLIC_URL`** в списке нет — используйте **TCP Proxy** в Postgres и соберите `postgresql://логин:пароль@хост_прокси:порт_прокси/railway`, как в **шаге 7.3**.  
   **Внутренний** `postgres.railway.internal` в `.env` на ПК **не подходит** — только публичная строка или прокси.
3. В тот же `.env` добавьте **`SEED_ADMIN_PASSWORD`** (и при желании `SEED_ADMIN_LOGIN`).
4. В терминале (PowerShell). Если при `npm` появляется ошибка про **`npm.ps1`** и **«выполнение сценариев отключено»** — используйте **`npm.cmd`** вместо `npm` (см. таблицу **«Типичные проблемы»** ниже) или командную строку **cmd.exe**.

   ```powershell
   cd c:\Sborka_Project\server
   npm.cmd install
   npm.cmd run db:seed
   ```

5. Должно появиться сообщение вроде **«Готово: пользователь admin»**. Если ошибка «connection refused» / «timeout» — в настройках Postgres на Railway проверьте, что разрешены **внешние подключения** / есть **public** URL (зависит от плана и настроек сети). Тогда используйте вариант 2.

**Вариант 2 — из облака Railway:** если в интерфейсе есть **Shell / One-off command** у сервиса с репозиторием, можно выполнить там `cd server` и `npm run db:seed` при тех же Variables (в т.ч. `DATABASE_URL` и `SEED_ADMIN_PASSWORD`). У новичков чаще срабатывает **вариант 1**.

### 7.7. Проверка после деплоя

Подставьте **ваш** домен приложения вместо примера `https://sborka-pro-production.up.railway.app`.

**1) Health**

В браузере откройте:

`https://ВАШ-ДОМЕН.up.railway.app/health`

В JSON должно быть **`"db": true`** и **`"phase": "B"`**. Если **`"db": false`** — приложение не видит базу: проверьте имя переменной **`DATABASE_URL`**, перезапуск деплоя, логи **Deploy**.

**2) Вход (получить токен)**

В PowerShell (подставьте свой URL, логин и пароль из сида).

**Важно:** в PowerShell после **`-d` обязательно возьмите тело в кавычки: `-d "$body"`**. Если написать **`-d $body` без кавычек**, JSON с пробелами разобьётся на несколько «слов», и `curl` воспримет **`https`** как имя хоста → ошибка **`Could not resolve host: https`**.

**Вариант A — `curl.exe` (обратите внимание на `"$body"`):**

```powershell
$base = "https://ВАШ-ДОМЕН.up.railway.app"
$body = '{"login":"ВАШ_ЛОГИН_ИЗ_SEED","password":"ВАШ_ПАРОЛЬ_ИЗ_SEED"}'
curl.exe -sS -X POST "$base/auth/login" -H "Content-Type: application/json" -d "$body"
```

**Вариант B — встроенный `Invoke-RestMethod`** (часто проще на Windows, к своему же серверу):

```powershell
$base = "https://ВАШ-ДОМЕН.up.railway.app"
Invoke-RestMethod -Uri "$base/auth/login" -Method Post -ContentType "application/json; charset=utf-8" -Body '{"login":"ВАШ_ЛОГИН_ИЗ_SEED","password":"ВАШ_ПАРОЛЬ_ИЗ_SEED"}'
```

**Вариант C — скрипт из репозитория (без ручного JSON в PowerShell):** в `server/.env` задайте **`AUTH_TEST_BASE`**, при необходимости **`AUTH_TEST_LOGIN`** и **`AUTH_TEST_PASSWORD`** (если не задать последние два — скрипт возьмёт **`SEED_ADMIN_LOGIN`** / **`SEED_ADMIN_PASSWORD`**). Затем из папки `server`: **`npm run auth:test`**.

В ответе будет JSON с полем **`token`**. Скопируйте значение `token` (длинная строка).

**3) Список заказов**

Сначала снова задайте **`$base`**, если открыли новое окно PowerShell:

```powershell
$base = "https://ВАШ-ДОМЕН.up.railway.app"
$token = "ВСТАВИТЬ_ТОКЕН_ИЗ_ШАГА_2"
curl.exe -sS "$base/orders" -H "Authorization: Bearer $token"
```

Должен вернуться JSON с массивом **`orders`**. Если **`moysklad_not_configured`** — в Variables приложения нет **`MOYSKLAD_TOKEN`** (тот же, что для вебхука). Если список пустой — в МойСклад нет заказов в статусе из **`MOYSKLAD_ASSEMBLY_STATE_NAME`** (по умолчанию «Сборка»).

### 7.8. Куда смотреть, если что-то не работает

| Симптом | Что сделать |
|--------|-------------|
| `/health` → `"db": false` | У сервиса приложения нет **`DATABASE_URL`** или неверная строка; база не запущена. |
| Деплой падает после добавления Postgres | Откройте **Deploy Logs** — часто опечатка в переменной или база ещё не готова; повторите деплой. |
| `db:seed` — ошибка подключения | Проверьте, что строка **`DATABASE_URL`** скопирована полностью; для запуска с ПК нужен доступ к Postgres из интернета (настройки Railway / public). |
| `401` / `invalid_credentials` | Логин в запросе должен совпадать с пользователем в БД (**тот `SEED_ADMIN_LOGIN`, с которым выполняли `db:seed`**; по умолчанию в примерах — `admin`, но у вас может быть другой). Пароль — тот, что был при сиде. Если меняли `SEED_*` и не перезапускали сид — в БД старые значения. Проверка: **`npm run auth:test`** в папке `server` (`AUTH_TEST_*` в `.env`, см. **`server/.env.example`**). |
| `503` «jwt_not_configured» | Нет **`JWT_SECRET`** или короче 32 символов. |
| `503` «database_unavailable» | Нет **`DATABASE_URL`** у приложения. |
| **`curl: (6) Could not resolve host: https`** в шаге 7.7 | В PowerShell после **`-d`** передайте JSON в кавычках: **`-d "$body"`**, либо используйте **`Invoke-RestMethod`** из шага 7.7. Частая причина — **`-d $body` без кавычек**. |

## Типичные проблемы

| Симптом | Что проверить |
|--------|----------------|
| Ошибка про `` ```powershell `` | В терминал вставляли кавычки из Markdown; вводите только команду (`git status` и т.д.). |
| `git` не распознан | Установите Git for Windows, перезапустите терминал, см. блок выше. |
| Промпт `->`, команды не выполняются | Часто в `git commit -m "текст` **нет закрывающей кавычки** `"`. Нажмите **Ctrl+C**, выполните коммит снова: `git commit -m "полное сообщение"` (кавычка в конце обязательна). |
| Ошибка / ничего не ушло на GitHub | Проверьте опечатку: нужно **`git push`**, не `it push` (без буквы **g**). После исправления снова: `git push -u origin main`. |
| Красная простыня в PowerShell про `It` и параметр `test` | Часто ввели **`it`** вместо **`git`** (например `it config` вместо `git config`). Исправьте на **`git`**. |
| **`git push` долго «молчит», нет строк Enumerating objects** | Обычно ждёт **авторизацию GitHub**. Проверьте: всплывающее окно **Git Credential Manager** (за другими окнами), иконка в **панели задач**, браузер для входа. Либо сеть/VPN блокирует github.com. Отмена: **Ctrl+C**. Пароль от аккаунта GitHub для HTTPS **не подходит** — нужен **Personal Access Token** (classic) с правом `repo`: [создание токена](https://github.com/settings/tokens). При запросе пароля вставьте токен. Повтор с подробным логом: `git push -u origin main --verbose`. |
| `Author identity unknown` | Выполните `git config --global user.name` и `user.email`, затем снова `git commit`. См. блок «Windows» выше, пункт 3. |
| Build failed: no package.json / Railpack «only documentation» | Укажите **Root Directory** = `server` или нажмите в диагнозе Railway **Set root directory**. |
| Приложение падает сразу | Логи **Deploy** и **HTTP**; убедитесь, что `npm start` запускается из `server/`. |
| 502 / сервис не отвечает | Сервис «спит» на бесплатном плане после простоя — откройте URL снова; проверьте лимиты плана Railway. |
| **`415 Unsupported Media Type`** при запросе к `api.moysklad.ru` | Часто виноват **Windows PowerShell 5.x** и `Invoke-RestMethod` (кодировка или потеря `Content-Type` для тела). Используйте шаг **Г1** (`npm run moysklad:register-webhooks`) или **Г2** (`curl.exe`, не алиас `curl`). |
| Команда **`curl`** в PowerShell ведёт себя странно | В PowerShell **`curl`** часто — алиас на **Invoke-WebRequest**, это не тот инструмент. Для МойСклад вызывайте **`curl.exe`** (как в шаге **Г2**). |
| МойСклад не доставляет webhook | URL только **https**; путь `/webhook/moysklad`; вебхук создан через **JSON API** (`POST /entity/webhook`), не через несуществующий пункт меню; тариф/права админа; таймаут **1,5 с** у МойСклад. |
| Нет пункта «Вебхуки» в настройках | Это нормально: см. **шаг 6** — регистрация только через API + токен из «Токены». |
| `/health` показывает `"db": false` после шага 7 | У **сервиса приложения** нет **`DATABASE_URL`** или ошибка подключения; см. **шаг 7.3–7.8**. |
| `npm run db:seed` не подключается к базе с ПК | Используйте **public** `DATABASE_URL`, либо сид из среды Railway; см. **шаг 7.6**. |
| **`npm.ps1`**, «выполнение сценариев отключено», `PSSecurityException` | В PowerShell по умолчанию вызывается обёртка **`npm.ps1`**. Используйте **`npm.cmd install`** и **`npm.cmd run …`** **или** один раз: `Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned` (подтвердите **Y**). **Или** откройте **cmd.exe** (не PowerShell) — там обычный `npm install` без этой ошибки. |
