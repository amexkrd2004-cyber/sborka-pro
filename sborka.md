Этот файл — **опорная точка для работы над проектом** и для ИИ-ассистента в новых сессиях.

## 1. О проекте

| Поле | Значение |
|------|----------|
| Название | СборкаПро (условное) |
| Назначение | Android-приложение для кладовщиков: сборка заказов, скан ШК, смена статусов в МойСклад, печать стикеров; веб-панель администратора; сервер (webhook + API). |
| Источник требований | `ТЗ_СборкаПро.docx` в корне репозитория |
| Интеграции | **МойСклад** (REST API, webhooks), **amoCRM** (синхронизация через виджет **amgroup**, уже настроена и работает) |

## 2. Что уже есть у заказчика (на старте)

- Учётные записи и рабочие процессы в **МойСклад** и **amoCRM**.
- **Интеграция МойСклад ↔ amoCRM** (amgroup) настроена и проверена в бою.

**Уже не «будущий» сервер:** webhook **МойСклад → Railway** настроен и проверен (`assemblyMatch` при статусе «Сборка»). **Ещё не считать готовым без отдельной настройки:** доп. поля заказа в МойСклад под ТЗ, тестовые кладовщики в приложении — см. фазы B–C.

---

## 3. Целевой стек (из ТЗ)

- Мобильное приложение: **React Native (Expo)**, Android APK (EAS Build).
- Сервер: **Node.js + Express** (Railway / Render / VPS).
- БД: **SQLite** (старт) или **PostgreSQL**.
- Push: **Expo Push Notifications**.
- Сканер: **expo-barcode-scanner**.
- Печать: **ESC/POS** (термопринтеры Bluetooth/Wi-Fi); Zebra ZPL — опционально позже.
- Веб-панель: **React SPA** на том же сервере (или отдельный деплой — уточнить при реализации).

### 3.1. Структура кода в репозитории

|------|------------|

Вход `server/src/index.js`, БД `server/src/db/`, webhook `server/src/routes/webhook.js`, auth `server/src/routes/auth.js`, заказы `server/src/routes/orders.js`, МойСклад `server/src/services/moyskladApi.js`, `moyskladWebhookWorker.js`, `moyskladOrders.js`. ### 5.1. Актуализация руководств (по мере роста проекта)

Сейчас в `docs/rukovodstva/*.md` много **нейтральных** формулировок («обычно», «вроде», «раздел вроде…») — это нормально для черновика. По мере появления реального UI, URL и процессов их нужно **заменять на конкретные**: точные названия кнопок, путей в меню, полей МойСклад, адресов.

**Правило:** любое изменение, которое видят пользователи или сисадмин при развёртывании, **в том же цикле** (тот же PR, та же задача или сразу следующим коммитом) отражается в соответствующем руководстве. Не копить «потом допишем».

**Кому что править (ориентир):**

| Изменили в проекте | Обновить |
|--------------------|----------|
| Экраны, сценарии, кнопки, push, сканер, печать в APK | `docs/rukovodstva/kladovshik.md` + скриншоты |
| Веб-панель: разделы, отказы, пользователи, отчёты, дашборд | `docs/rukovodstva/administrator.md` + скриншоты |
| Env, деплой, webhook, БД, EAS, домены, команды, восстановление | `docs/rukovodstva/systemnyy-administrator.md` |
| План фаз, стек, важные решения | `sborka.md` (и при необходимости — все руководства) |

**Версия документа:** внизу каждого руководства указаны *Версия документа* и статус. При существенной правке поднять минорную версию (0.1 → 0.2) и кратко описать в журнале ниже.

**Для сессий с ИИ:** в конце задачи можно явно попросить: *«Актуализируй руководства по разделу 5.1 в `sborka.md`»* — тогда правки войдут в `docs/rukovodstva/` и при необходимости в журнал раздела 7.
